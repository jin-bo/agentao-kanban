# v0.1.1 Improvement Plan

## Goal

`v0.1.1` does not change the core scheduler/runtime architecture. The focus is
to turn the existing internal capabilities into an operator-friendly workflow:

- maintain cards without hand-editing Markdown
- recover blocked cards in a standard way
- inspect execution events and raw traces from the CLI
- detect common board integrity issues

Out of scope for this version:

- multi-card concurrency
- ACP / remote workers
- UI / web frontend
- changing the single-writer orchestrator model

## Why This Version

The current repository already has:

- a stable orchestrator boundary
- persisted board state and execution events
- raw transcript retention
- role-routed multi-agent execution
- healthy automated tests

What is still missing is the operator surface. `v0.1.1` closes that gap before
the project takes on more runtime complexity.

## Global Rules

These apply to every new write command introduced in this version. Each
subsection below refers back to them rather than restating.

- **Daemon lock**: every write command goes through `_require_writable`
  (`cli.py:117`) and refuses to mutate while `.daemon.lock` is held, unless
  `--force` is passed. This includes `card edit`, `card context *`,
  `card acceptance *`, and `requeue`.
- **History tagging**: CLI-written history entries use the `[system]` tag,
  identical to `move` / `block` / `unblock`. Only agent-driven entries carry
  a role tag like `[worker]`.
- **Read-only commands** (`events`, `traces`, `doctor`) do not require the
  lock and do not mutate the board.

## Day 0 Prerequisites

These are correctness gaps in the current codebase that `v0.1.1` surfaces
(by exposing more write paths). They must land before or alongside the
first new write command.

### 0.1. `ContextRef.coerce` must soft-fail on malformed entries

Current state (`kanban/models.py`): `coerce` raises `KeyError` on a dict
missing `path` and `TypeError` on non-string/non-dict/non-`ContextRef`
values. Combined with `_card_from_toml_dict` calling it eagerly, a single
malformed ref aborts the whole card load — defeating the resilience added
in the most recent patch.

Required change:

- make `coerce` return `None` (or raise a dedicated `InvalidContextRef`)
  for unrecognizable shapes
- in `_card_from_toml_dict`, drop `None` results and log a warning per
  skipped ref, keeping the rest of the card loadable
- CLI writes still validate strictly (see §Data Changes 2)

Without this, `doctor` can never report "invalid context ref" — the card
never reaches doctor at all.

### 0.2. Store API: cross-card event listing

`BoardStore` today only exposes `events_for_card(card_id)`. `kanban events`
(no id) requires a new helper:

- `list_events(*, limit: int | None = None) -> list[CardEvent]`
- `list_execution_events(*, card_id=None, role=None, limit=None) -> list[ExecutionEvent]`

CLI must not read `events.log` directly.

## Target Commands

### 1. `kanban card edit`

Purpose: update existing cards without editing board Markdown by hand.

Target shapes:

```bash
kanban card edit <card_id> --title "New title"
kanban card edit <card_id> --goal "New goal"
kanban card edit <card_id> --priority HIGH
kanban card edit <card_id> --set-status review
kanban card edit <card_id> --blocked-reason "needs dataset"
kanban card edit <card_id> --clear-blocked-reason
```

Rules:

- `--set-status` is an operator override, not normal workflow progression,
  and is intentionally restricted to statuses that have no `owner_role`
  expectation: `inbox`, `ready`, `blocked`, `done`. Setting `doing` /
  `review` / `verify` directly would desync status and `owner_role`, so
  it is disallowed — use `requeue` for recovery.
- `--set-status blocked` requires `--blocked-reason` in the same call
  (otherwise we create exactly the inconsistency `doctor` is supposed to
  detect).
- `--set-status` always resets `owner_role` to `None` unless the new
  status is `review` or `verify` (which are disallowed — see above), so
  no explicit owner-role reset flag is needed.
- manual status changes must append a clear `[system]` history entry.

Expected history notes:

- `Manual edit via CLI`
- `Status manually set to <status> via CLI`
- `Blocked reason updated via CLI`
- `Blocked reason cleared via CLI`

### 2. `kanban card context add|rm|list`

Purpose: make `context_refs` a first-class operator input.

Target shapes:

```bash
kanban card context list <card_id>
kanban card context add <card_id> --path docs/api.md --kind required --note "API contract"
kanban card context rm <card_id> --path docs/api.md
```

Rules:

- `kind` accepts only `required` or `optional`
- path uniqueness is per card
- repeated `add` on the same path should upsert rather than duplicate
- output order should be stable and readable

Expected history notes:

- `Context added: docs/api.md [required]`
- `Context updated: docs/api.md [optional]`
- `Context removed: docs/api.md`

### 3. `kanban card acceptance add|rm|list|clear`

Purpose: let humans refine planner-produced acceptance criteria.

Target shapes:

```bash
kanban card acceptance list <card_id>
kanban card acceptance add <card_id> --item "Generate workspace/reports/summary.md"
kanban card acceptance rm <card_id> --index 2
kanban card acceptance clear <card_id>
```

Rules:

- remove by index, not by text
- invalid indexes must fail with a clear message
- all changes append history entries

Expected history notes:

- `Acceptance criterion added: Generate workspace/reports/summary.md`
- `Acceptance criterion removed at index 2`
- `Acceptance criteria cleared via CLI`

### 4. `kanban events`

Purpose: inspect `events.log` without opening files manually.

Target shapes:

```bash
kanban events
kanban events <card_id>
kanban events <card_id> --role planner
kanban events <card_id> --limit 20
kanban events <card_id> --json
```

Expected event fields (execution events have all of them; plain events
only have `at` / `card_id` / `message`):

- `at`
- `card_id`
- `role` *(execution events only)*
- `prompt_version` *(execution events only)*
- `duration_ms` *(execution events only)*
- `attempt` *(execution events only)*
- `message`
- `raw_path` *(execution events only, when retention is on)*

Rules:

- two record shapes coexist in `events.log`: plain events
  (`append_event`) and execution events (`append_execution_event`).
  Both must render.
- `--role <r>` filters to execution events matching that role; plain
  events (status-change, lock, manual edits) are **excluded** while
  `--role` is active. If an operator wants to see system events
  interleaved, they can rerun without `--role`.
- default output is human-readable (one event per line, tab- or
  column-aligned); `--json` produces one JSON record per line.
- listing across all cards (no `card_id` positional) is supported and
  paged by `--limit` (default: latest 50).

### 5. `kanban traces`

Purpose: surface retained raw transcripts from `workspace/raw/`.

Target shapes:

```bash
kanban traces <card_id>
kanban traces <card_id> --role worker
kanban traces <card_id> --latest
```

Version scope:

- list-only is sufficient for `v0.1.1`
- transcript preview/rendering can wait for a later version

Displayed fields:

- role
- timestamp
- path
- size

Rules:

- if `workspace/raw/<card-id>/` does not exist (board has only run with
  the mock executor, or retention is off), print a one-line message
  (`no traces retained for <card_id>`) and exit 0 — not an error.
- `--latest` without `--role` returns the most recent transcript across
  all roles.

### 6. `kanban requeue`

Purpose: define the standard recovery path from `BLOCKED` back into flow.

Target shapes:

```bash
kanban requeue <card_id>
kanban requeue <card_id> --to inbox
kanban requeue <card_id> --to ready --note "added missing context"
```

Default behavior:

- if current status is `blocked`, move to `inbox`
- clear `blocked_reason`
- append a system history note

Expected history note:

- `Requeued from blocked to inbox: added missing context`

Open choice:

- non-blocked cards may either be rejected or allowed as an operator override
- preferred behavior for `v0.1.1`: allow, but log the previous state clearly

### 7. `kanban doctor`

Purpose: run basic board integrity and workflow health checks.

Target shape:

```bash
kanban doctor
kanban doctor --json
```

Initial checks:

- dependency points to a missing card *(error)*
- card is `blocked` but `blocked_reason` is empty — **defensive only**,
  since the orchestrator and CLI never produce this state; catches hand-
  edited or corrupted cards *(warning)*
- card is `done` but `outputs.verification` is empty (`None`, `""`, or
  `{}`). Accept either string (legacy / mock) or dict (structured) — do
  not require the new `{status: "verified", ...}` shape, else mock-run
  boards light up red *(warning)*
- card is in `review` or `verify` but the upstream role's output key is
  missing from `outputs` (`review` needs `implementation`; `verify`
  needs `implementation` and `review`) *(error)*
- `context_refs.kind` is not in `{"required", "optional"}` *(warning)*
- card file was skipped by the load path (see Day 0 prereqs) *(error)*

Exit codes:

- `0`: no issues
- `1`: warnings only
- `2`: at least one error

`--json` output schema (frozen for this version):

```json
{"checks": [
  {"severity": "error|warning", "rule": "dep-missing",
   "card_id": "...", "message": "..."}
]}
```

Extra keys may be added in later versions; `severity`, `rule`,
`card_id`, `message` are the stable contract.

## Data Changes

### 1. `Card` schema is frozen in `v0.1.1`

No new fields. Spend the version budget on command surface and
diagnostics.

### 2. `ContextRef` validation: strict on write, soft on load

Two call sites, two contracts:

- **CLI write path** (`card context add`): reject invalid `kind` with a
  clear error, reject malformed dicts, reject empty `path`. Never store
  garbage.
- **Load path** (`_card_from_toml_dict` → `ContextRef.coerce`): `coerce`
  returns `None` for unrecognizable shapes; caller drops `None` entries
  and logs one warning per skipped ref. The rest of the card loads.

This split — strict ingress, lenient recovery — is what lets `doctor`
see and report invalid refs without the load path killing the card
first. See Day 0 §0.1.

### 3. Do not change the `events.log` write schema

Current execution event records are already sufficient for `v0.1.1`.

Planned change:

- add read/query helpers over existing JSONL records
- support filtering by `card_id`, `role`, and `limit`

Non-goal:

- no event schema migration in this version

### 4. Normalize history entries from CLI mutations

All operator-driven mutations should append predictable history messages so
future tooling can analyze them mechanically.

Examples:

- `Manual edit via CLI`
- `Context added: ...`
- `Context removed: ...`
- `Acceptance criterion added: ...`
- `Acceptance criterion removed at index ...`
- `Requeued from blocked to ...`

### 5. Add read helpers in the markdown store

Recommended store-level helpers:

- list execution events by card / role / limit
- list raw transcripts by card / role

The CLI should not parse `events.log` and `workspace/raw/` directly.

## Test Checklist

### 0. Day 0 prerequisites

- a card with a malformed `context_refs` entry (missing `path`, wrong
  type, invalid `kind`) loads with the bad ref dropped and a warning
  logged; other refs survive
- `store.list_events(limit=N)` returns the most recent N events across
  all cards, mixed shapes
- `store.list_execution_events(card_id=X, role=worker)` filters to
  execution-event records only, matching role

### 1. `card edit`

- edits `title`
- edits `goal`
- edits `priority`
- sets status manually and appends history
- sets blocked reason
- clears blocked reason
- refuses mutation while daemon lock is held (one case suffices;
  §Global Rules guarantees the same for every write command)
- rejects `--set-status doing`/`review`/`verify`
- rejects `--set-status blocked` without `--blocked-reason`

### 2. `card context`

- adds a required context ref
- adds a note-bearing context ref
- upserts when the same path is added twice
- removes by path
- lists context refs in stable order
- rejects invalid `kind`
- round-trips through reload

### 3. `card acceptance`

- adds a criterion
- removes by valid index
- rejects invalid index
- clears all criteria
- round-trips through reload

### 4. `events`

- lists recent events across the board
- filters to one card
- filters by role
- applies limit
- `--json` output parses cleanly
- legacy plain events and execution events can coexist

### 5. `traces`

- lists raw transcripts when present
- filters by role
- respects retention outcomes
- prints a clear message when no traces exist

### 6. `requeue`

- blocked card requeues to `inbox` by default
- `--to ready` works
- clears `blocked_reason`
- appends a recovery history note
- behavior on non-blocked cards is explicit and tested

### 7. `doctor`

- reports missing dependency targets (error)
- reports blocked card without reason (warning; defensive, seed the
  case by hand-crafting a card file)
- reports done card with empty `outputs.verification` (warning);
  **does not** flag `outputs.verification = "Acceptance criteria verified."`
  (legacy string form) as missing
- reports `review`/`verify` card with missing upstream outputs (error)
- reports invalid `context_refs.kind` (warning)
- `--json` output matches the frozen schema (`severity`, `rule`,
  `card_id`, `message`)
- returns success when the board is healthy

### 8. Regression Coverage

- all existing tests continue to pass
- markdown board compatibility does not regress
- multi-agent executor behavior is unchanged
- daemon lock behavior is unchanged

## Release Criteria

`v0.1.1` is complete when:

- Day 0 prerequisites are merged and verified by tests
- routine card maintenance no longer requires manual Markdown edits
- blocked cards can be resumed through a standard CLI path
- operators can inspect what happened to a card from the CLI
- retained raw traces are discoverable
- board integrity problems are detectable by `doctor`
- all added behavior is covered by tests
