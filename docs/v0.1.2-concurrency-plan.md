# v0.1.2 Concurrency Kernel Plan

## Goal

`v0.1.2` upgrades `kanban` from a single-process dispatcher into a
concurrency-capable local runtime while preserving the core invariant:

- `KanbanOrchestrator` remains the only component that commits card state transitions.

This version is about execution concurrency, not product surface.

Primary goals:

- replace the board-wide execution guard with per-card execution locks + leases
- support multiple local workers pulling from a shared queue
- recover from stuck or crashed executions
- record enough runtime metadata to debug retries, timeouts, and resource usage

Out of scope:

- web UI / dashboard
- remote workers / ACP transport
- GitHub PR workflow
- collaborative permissions / multi-user semantics
- container or worktree isolation beyond the minimum hooks needed for a later version

## Why This Version

The current repository already has:

- a stable scheduler boundary in `KanbanOrchestrator`
- a local daemon loop
- persisted board state and append-only events
- role-routed execution with failure normalized to `BLOCKED`
- test coverage over scheduler and daemon basics

What it does not have is safe parallel execution. Today, the daemon lock
prevents concurrent mutation at the board level, but it also prevents useful
parallelism. `v0.1.2` should add concurrency without relaxing workflow
authority.

## Core Invariants

These rules are the design guardrails for the entire version.

- `KanbanOrchestrator` is still the only state-transition authority. Workers may execute tasks in parallel, but they do not move cards between workflow statuses on their own.
- At most one live execution claim may exist per card at any time.
- Queue state and execution claims are advisory runtime records; board card files remain the source of truth.
- A crashed worker must never be able to leave a card permanently un-runnable.
- Recovery must be explicit and observable. Timeouts, retries, lease steals, and stale claim cleanup all emit structured events.
- A worker may reserve work, execute work, emit heartbeats, and submit a result; it may not directly mutate a card file's workflow status outside the orchestrator-owned commit path.

## Target Runtime Model

`v0.1.2` introduces a split between three responsibilities:

1. `Scheduler`
   The orchestrator scans board state, picks actionable cards, and creates execution claims.

2. `Workers`
   One or more daemon worker loops claim executable work, run the executor, heartbeat the lease, and submit a result envelope.

3. `Committer`
   A single orchestrator-owned commit path validates the result envelope, verifies that the lease is still valid, and only then writes card updates and status transitions.

This is intentionally not a distributed queueing system. It is a local,
filesystem-backed concurrency model with single-writer state commits.

## State Machine Changes

The workflow statuses remain:

- `inbox`
- `ready`
- `doing`
- `review`
- `verify`
- `done`
- `blocked`

`v0.1.2` does not add a new user-facing workflow column like `running` or
`claimed`. Instead, runtime execution state lives beside the board in queue /
lease metadata.

That separation is important:

- workflow status answers "where is this card in delivery?"
- lease / claim state answers "is there an in-flight execution for this card right now?"

## Filesystem Layout

Add a runtime area under `workspace/board/runtime/`:

```text
workspace/board/
  cards/
  events.log
  runtime/
    queue.jsonl
    claims/
      <card-id>.json
    results/
      <card-id>-<attempt>.json
    workers/
      <worker-id>.json
```

Rules:

- `claims/<card-id>.json` is the single live claim record for a card.
- `results/` contains submitted execution result envelopes waiting to be committed.
- `workers/<worker-id>.json` is a heartbeat presence file for observability only.
- `queue.jsonl` is an append-only scheduler audit log, not the source of truth for what is runnable.

The scheduler still derives runnable work from board state; the queue log is
used to explain which worker claimed what and when.

## Per-Card Lock and Lease Model

### Claim Record

Each live claim record should include:

- `card_id`
- `role`
- `status_at_claim`
- `worker_id`
- `claim_id`
- `attempt`
- `claimed_at`
- `lease_expires_at`
- `heartbeat_at`
- `timeout_s`
- `retry_count`

Storage shape:

- one JSON file per card at `runtime/claims/<card-id>.json`
- written with atomic temp-file + `os.replace()`

### Lock Semantics

Lock acquisition is per card, not per board:

- the scheduler may only create a claim if no valid live claim exists for that card
- multiple workers may hold claims concurrently for different cards
- a worker renewing a lease must prove claim ownership via `claim_id`

The old board-wide `.daemon.lock` should remain only as a compatibility guard
for legacy write commands in `v0.1.2`; it should no longer be the mechanism that
serializes execution.

### Lease Semantics

Recommended defaults:

- lease duration: `60s`
- heartbeat interval: `15s`
- execution timeout: role-specific, default `10m`
- stale-claim recovery threshold: `lease_expires_at < now`

A claim is considered recoverable when:

- the worker heartbeat file is stale, or
- the claim lease has expired

Recovery action:

- mark the attempt failed in events
- either requeue for retry or move to `BLOCKED`, according to retry policy

## Queue and Worker Design

### Scheduler Loop

The scheduler daemon remains responsible for:

- scanning board state
- finding actionable cards
- skipping cards with valid live claims
- creating new claims up to a configured concurrency limit
- recording `execution.claimed` events

It does not run the executor inline anymore when worker mode is enabled.

### Worker Loop

Each worker daemon:

- advertises presence via `runtime/workers/<worker-id>.json`
- polls for claims assigned to itself or unassigned claims, depending on mode
- starts execution for one claimed card
- heartbeats while execution is in progress
- writes a result envelope to `runtime/results/`
- emits `execution.started`, `execution.heartbeat`, `execution.finished`, or `execution.failed`

Recommended initial modes:

- `kanban daemon scheduler`
- `kanban daemon worker`
- `kanban daemon all`

`all` runs one scheduler thread plus one worker loop in the same process for
local convenience. Production semantics are still local-only.

### Concurrency Limit

Add two explicit limits:

- `scheduler.max_claims`: max concurrent claims across the board
- `worker.max_parallelism`: max executions a worker process may run simultaneously

`v0.1.2` should start conservatively:

- one execution thread per worker process
- multiple worker processes allowed

No need for in-process thread pools until the file/lease model proves stable.

## Single-Writer Commit Path

This is the most important correctness rule in the version.

Workers do not call `store.move_card()` after execution. Instead they submit:

- `claim_id`
- `card_id`
- `role`
- `attempt`
- `started_at`
- `finished_at`
- `duration_ms`
- `executor_result`
- `resource_usage`
- `failure_reason` if any

The scheduler / committer then:

1. loads the current card
2. loads the current claim
3. validates `claim_id` and `attempt`
4. rejects stale or duplicate results
5. calls orchestrator-owned apply logic
6. clears the live claim
7. emits commit events

This preserves single-writer state ownership even though execution is parallel.

## Timeout, Retry, and Stuck Task Policy

### Timeout Policy

Each role should gain a configurable timeout:

- planner: short
- reviewer / verifier: medium
- worker: longest

Suggested defaults:

- planner: `120s`
- reviewer: `300s`
- verifier: `300s`
- worker: `1800s`

Timeout handling:

- worker notices local timeout and submits a failed result envelope when possible
- scheduler also performs timeout enforcement when it sees an expired lease

### Retry Policy

Add explicit retry metadata per execution attempt:

- `attempt`
- `retry_of_claim_id`
- `retry_reason`

Recommended initial policy:

- infrastructure failure: retry up to `2` times
- lease expiry / worker crash: retry up to `1` time
- malformed agent response: no automatic retry
- reviewer/verifier functional rejection: no automatic retry; move to `BLOCKED`

This keeps retries focused on runtime failures, not task-quality failures.

### Stuck Task Recovery

Cases to recover:

- worker process dies mid-run
- host is suspended and heartbeat stops
- claim file exists but worker is gone
- result file exists but commit path crashed before applying

Required recovery rules:

- stale claim without result: mark attempt failed, clear claim, retry or block
- result file with matching live claim: commit on next scheduler pass
- result file with missing / replaced claim: mark as orphaned result, keep for audit, do not apply

## Event Model Upgrade

The current event model records plain events and execution summaries. `v0.1.2`
needs runtime lifecycle events.

### New Event Categories

- `execution.claimed`
- `execution.started`
- `execution.heartbeat`
- `execution.finished`
- `execution.failed`
- `execution.timed_out`
- `execution.retried`
- `execution.claim_recovered`
- `execution.result_orphaned`
- `worker.started`
- `worker.stopped`

### Required Fields

Every execution event should be able to carry:

- `event_type`
- `card_id`
- `claim_id`
- `attempt`
- `worker_id`
- `role`
- `status_before`
- `status_after` when applicable
- `at`
- `duration_ms` when applicable
- `failure_reason` when applicable
- `retry_of_claim_id` when applicable
- `lease_expires_at` when applicable
- `heartbeat_at` when applicable
- `prompt_version` when applicable
- `raw_path` when applicable
- `resource_usage` when available

### Resource Usage

Do not overdesign this. Start with lightweight local metrics:

- `pid`
- `rss_bytes`
- `cpu_seconds`
- optional `workdir_size_bytes`

The resource usage payload may be absent for some executors. Missing values are
acceptable; missing fields should not block event emission.

## CLI and Operator Surface

Add runtime inspection commands instead of overloading `events`:

```bash
uv run kanban claims
uv run kanban claims <card_id>
uv run kanban workers
uv run kanban queue
uv run kanban retry <card_id>
uv run kanban recover --stale
```

Minimum requirements for `v0.1.2`:

- `claims` shows active claims with lease age and owner worker
- `workers` shows active workers with last heartbeat
- `recover --stale` runs stale claim cleanup once

`retry` can be deferred if automatic retry plus `requeue` is sufficient.

## Data Model Changes

### 1. Execution Metadata Type

Add a structured execution metadata object rather than further stretching
`CardEvent`.

Suggested additions:

- `ExecutionClaim`
- `ExecutionResultEnvelope`
- `WorkerPresence`

### 2. Card Outputs

Do not put runtime claim state into `Card.outputs`.

`Card.outputs` should remain about deliverables:

- planner notes
- implementation artifacts
- review findings
- verification evidence

Runtime control state belongs in `runtime/claims` and `events.log`.

### 3. Store Boundary

The markdown board store should gain helpers for:

- claim create / renew / clear
- stale claim enumeration
- result envelope write / read / clear
- worker presence heartbeat
- richer event append / query

These APIs should live beside the store, not inside the executor.

## Daemon Topology

`v0.1.2` should support these topologies:

1. Single process, combined mode
   One process runs scheduler + worker for local development.

2. One scheduler, many workers
   The scheduler claims work and commits results; workers execute cards.

3. Recovery-only run
   A one-shot command scans stale claims and pending results, then exits.

The scheduler count must remain exactly one per board.

Enforcement options:

- keep `.daemon.lock` for the scheduler role only
- workers do not take the board-global lock

That gives:

- exactly one scheduler / committer
- many workers
- per-card concurrency

## Failure Policy

Recommended normalized outcomes:

- claim acquisition conflict: no-op, emit debug log only
- worker startup failure: no card mutation, emit `worker.failed`
- executor exception: failed result envelope, eligible for retry policy
- malformed result envelope: `BLOCKED`
- commit validation mismatch: orphan result, emit `execution.result_orphaned`
- max retries exceeded: move card to `BLOCKED` with explicit runtime reason

The blocking reason should include the last runtime failure class, not just a
human summary. Example:

- `runtime timeout after 1800s on worker worker-3 (attempt 2/2)`

## Testing Plan

`v0.1.2` needs heavier concurrency testing than previous versions.

Required test classes:

- claim acquisition race: two workers try to claim the same card
- stale lease recovery: expired claim becomes runnable again
- duplicate result rejection: old result envelope does not overwrite newer state
- scheduler single-writer rule: only one process commits transitions
- retry chain audit: retries emit linked events with stable attempt numbering
- worker crash simulation: claim expires and is recovered
- recovery after restart: pending results commit after daemon restart

The existing deterministic mock executor should remain the primary harness for
these tests.

## Implementation Order

### Phase 0: Design Guardrails

- document claim / result / worker presence formats
- define event schema changes
- define retry matrix by failure type

### Phase 1: Runtime Storage

- add `runtime/` layout
- add claim CRUD + lease renewal helpers
- add result envelope storage
- add worker presence heartbeats

### Phase 2: Scheduler / Worker Split

- split current daemon into scheduler and worker roles
- keep a combined mode for local development
- preserve the existing serial mode behind a compatibility flag until tests pass

### Phase 3: Recovery Logic

- stale claim scavenger
- timeout enforcement
- retry counter and retry event chain
- orphan result detection

### Phase 4: Operator Commands

- `claims`
- `workers`
- `recover --stale`

## Milestones

Translate the implementation order into merge-sized milestones:

### M0. Runtime Types and Store Surface

Deliverables:

- new runtime dataclasses in `kanban/models.py`
- runtime helper methods in `kanban/store.py` protocol
- markdown runtime storage implementation in `kanban/store_markdown.py`
- tests for claim create / renew / clear and result envelope persistence

Exit criteria:

- runtime files are written atomically
- stale and live claims are distinguishable
- result envelopes round-trip through disk without loss

### M1. Scheduler / Worker Role Split

Deliverables:

- split current daemon into scheduler-only and worker-only loops
- combined `all` mode for local debugging
- scheduler-global lock retained only for scheduler / committer
- worker presence heartbeats

Exit criteria:

- one scheduler and two workers can run against one board
- two workers can process two different cards concurrently
- duplicate claim on the same card is impossible in tests

### M2. Commit Path and Recovery

Deliverables:

- result submission flow
- single-writer commit path
- stale claim scavenger
- orphan result handling

Exit criteria:

- worker writes no workflow status transitions directly
- scheduler can recover from restart and commit pending results
- expired claims can be retried or blocked deterministically

### M3. Retry / Timeout / Runtime Events

Deliverables:

- role timeout config
- retry matrix implementation
- richer execution event records
- `kanban events` output upgraded for runtime lifecycle records

Exit criteria:

- every retry chain is queryable by claim id / attempt
- timeout failures include role, worker, duration, and retry status
- operator can distinguish infrastructure failure from functional rejection

### M4. Operator Commands

Deliverables:

- `kanban claims`
- `kanban workers`
- `kanban recover --stale`
- optionally `kanban queue`

Exit criteria:

- operator can inspect all active claims and workers without reading raw files
- stale recovery can be triggered explicitly from CLI

## Proposed API Surface

The store protocol should gain explicit runtime helpers instead of exposing
runtime internals ad hoc:

```python
def create_claim(self, claim: ExecutionClaim) -> ExecutionClaim: ...
def get_claim(self, card_id: str) -> ExecutionClaim | None: ...
def renew_claim(
    self,
    card_id: str,
    *,
    claim_id: str,
    heartbeat_at: datetime,
    lease_expires_at: datetime,
) -> ExecutionClaim: ...
def clear_claim(self, card_id: str, *, claim_id: str | None = None) -> None: ...
def list_claims(self) -> list[ExecutionClaim]: ...
def list_stale_claims(self, *, now: datetime) -> list[ExecutionClaim]: ...

def write_result(self, result: ExecutionResultEnvelope) -> Path: ...
def read_results(self, *, card_id: str | None = None) -> list[ExecutionResultEnvelope]: ...
def delete_result(self, card_id: str, attempt: int) -> None: ...

def heartbeat_worker(self, presence: WorkerPresence) -> WorkerPresence: ...
def list_workers(self) -> list[WorkerPresence]: ...
```

The orchestrator should gain a commit-oriented method:

```python
def apply_execution_result(
    self,
    card_id: str,
    result: AgentResult,
    *,
    expected_role: AgentRole,
    expected_status: CardStatus,
) -> Card:
    ...
```

That separates "pick work" from "apply result" and avoids overloading `tick()`
with both responsibilities once workers become asynchronous.

## PR Breakdown

Recommended merge sequence:

1. `PR1: runtime dataclasses + store runtime helpers`
2. `PR2: scheduler/worker daemon split without retries`
3. `PR3: result envelopes + single-writer commit path`
4. `PR4: stale recovery + timeout enforcement + retries`
5. `PR5: runtime CLI commands + event output polish`

Each PR should leave the repo runnable. Avoid landing queue claims and async
worker loops in the same PR that introduces retry and orphan-result recovery.

## Open Questions

These should be decided before implementation starts, not mid-PR:

- Should a scheduler create unassigned claims, or assign `worker_id` at claim time?
- Should result envelopes embed the full `AgentResult`, or a flattened normalized shape?
- Should lease recovery steal an existing claim in place, or clear and create a new claim id?
- Should `execution.heartbeat` be persisted every time, or sampled to avoid event volume?
- Do we want a hidden compatibility mode that preserves the old serial daemon through `v0.1.2` rollout?

My recommendation:

- scheduler creates unassigned claims
- result envelopes store a normalized copy of `AgentResult`
- stale recovery creates a new claim id rather than stealing in place
- heartbeat events should be sampled or emitted only on significant age intervals
- keep a temporary serial mode behind `kanban daemon --legacy-serial` until M2 is stable

## Acceptance Criteria

`v0.1.2` is done when all of these are true:

- multiple worker daemons can execute different cards concurrently on one board
- no two workers can hold a valid live claim for the same card
- only the scheduler / committer writes workflow status transitions
- stuck executions are detected and recovered without manual file surgery
- retries and timeout outcomes are visible in structured events
- the board remains loadable and auditable after scheduler or worker crashes
- the test suite covers race, recovery, and duplicate-result scenarios

## Explicit Non-Goals

To keep this version narrow, do not include:

- browser UI for runtime monitoring
- remote transport or network queueing
- multi-machine coordination
- semantic auto-replan after failure
- branch / worktree isolation as a hard requirement for every executor

Those belong in later versions once the local concurrency kernel is stable.
